# Contributing

1. Fork the repository and create a feature branch from `main`.
2. Keep each pull request focused on one change set.
3. Follow the existing code and config patterns already used in the repo.
4. Run the relevant test suite before opening a PR.
5. Document user-facing changes in `README.md` or supporting docs when needed.
6. Avoid committing secrets, tokens, or machine-specific credentials.
7. Open a pull request with a clear summary, rationale, and testing notes.
8. Tests must pass before a PR will be considered ready for review.
